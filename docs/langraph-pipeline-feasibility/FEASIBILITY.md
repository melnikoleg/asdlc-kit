# Отчёт о целесообразности: пайплайн на LangGraph + вызов Claude Code через CLI

**Дата:** 2026-06-24
**Объект анализа:** `asdlc-kit` — агентный SDLC-набор (Agentic SDLC Kit v2)
**Вопрос:** стоит ли переносить текущий пайплайн на LangGraph, оркеструя агентов через вызовы Claude Code CLI.

---

## 1. Что представляет собой текущий сетап

Это **100% нативная реализация на Claude Code**, без единой строки кода оркестрации. Вся логика выражена декларативно через примитивы платформы:

| Слой | Реализация | Назначение |
|------|-----------|-----------|
| **Оркестрация** | `orchestrator-agent` + skill `sdlc-orchestrate` | Делегирует фазы через инструмент `Agent` |
| **Агенты (8)** | `.claude/agents/*.md` (Markdown + YAML-frontmatter) | product, planner, architect, developer, reviewer, qa, devops, orchestrator |
| **Машина состояний** | `docs/{issue}/STATE.json` | phase / iteration / failing_agents / artifacts |
| **Команды (19 skills)** | `.claude/skills/*/SKILL.md` | Точки входа `/factory`, `/sdlc-*` |
| **Guardrails** | 8 hooks (`PreToolUse`/`PostToolUse`/`Stop`/`SubagentStop`) | Блок деструктивных команд, защита секретов, валидация артефактов |
| **Контракты/правила** | `.claude/rules/*.md` + `CLAUDE.md` | Always-active инструкции, контракты артефактов |
| **Права** | `settings.json` allow/deny-листы | Песочница для Bash |

**Ключевые свойства текущей модели:**
- Состояние пайплайна — единый разделяемый контекст одной сессии Claude Code; субагенты получают свежий контекст и возвращают JSON-вердикт.
- Параллельная фаза review = три субагента (reviewer/qa/devops) запускаются одновременно через `Agent`.
- Fix-loop с лимитом 3 итерации + эскалация — описан текстом в skill, исполняется моделью.
- Human-approval gate реализован как «жёсткий STOP» в инструкции skill.
- Hooks дают **детерминированное** (не-LLM) обеспечение инвариантов безопасности.

## 2. Что значит «пайплайн на LangGraph + Claude Code через CLI»

Два независимых архитектурных решения, которые стоит оценивать раздельно:

### 2.1. LangGraph как движок оркестрации
Пайплайн описывается как **граф** (узлы = фазы/агенты, рёбра = переходы) на Python/TS. Состояние — типизированный `StateGraph` (вместо `STATE.json`). Fix-loop = цикл с условным ребром, approval gate = `interrupt()` + checkpointer, параллельный review = fan-out/fan-in узлы.

### 2.2. Claude Code как исполнитель узла (через CLI / SDK)
Каждый узел графа вызывает Claude Code в headless-режиме. Варианты вызова:

```bash
# Сырой CLI (subprocess)
claude -p "<prompt>" \
  --output-format stream-json \
  --append-system-prompt "$(cat .claude/agents/developer-agent.md)" \
  --allowedTools "Read,Write,Edit,Bash,Glob,Grep" \
  --permission-mode acceptEdits
```

**Важно:** для интеграции с Python-кодом LangGraph правильнее использовать **Claude Agent SDK** (`claude-agent-sdk`, Python/TS), а не парсить stdout сырого CLI. SDK даёт стриминг, управление сессиями, hooks, MCP и определения субагентов программно — это «CLI под капотом», но без хрупкого разбора текста.

## 3. Сравнение архитектур

| Критерий | Сейчас (нативный Claude Code) | LangGraph + CLI/SDK |
|----------|-------------------------------|---------------------|
| **Код оркестрации** | 0 строк (декларативно) | Граф + узлы + редьюсеры состояния (Python) |
| **Состояние** | `STATE.json` + контекст сессии | Типизированный `StateGraph` + checkpointer (SQLite/Postgres) |
| **Параллелизм** | Субагенты через `Agent` | `asyncio` fan-out, явный контроль |
| **Approval gate** | Текстовый STOP в skill | `interrupt()` — детерминированная пауза/возобновление |
| **Fix-loop / лимиты** | Инструкция модели (мягкий контроль) | Код-цикл (жёсткая гарантия `iteration < 3`) |
| **Guardrails** | Hooks (детерминированы) | Сохраняются как hooks Claude Code **или** дублируются в Python |
| **Наблюдаемость** | Audit-log хуки, ручной разбор | LangSmith-трейсинг, метрики токенов/латентности на узел |
| **Воспроизводимость** | Зависит от сессии | Checkpoint → точный replay/resume с любого узла |
| **Возобновление после сбоя** | `--resume` из `STATE.json` | Durable execution из checkpointer |
| **Портируемость движка** | Только внутри Claude Code | LLM-агностично (можно менять провайдера на узле) |
| **Барьер входа** | Markdown, без сборки | Python-проект, зависимости, тесты, деплой |
| **Стоимость вызова** | 1 процесс/сессия | N запусков CLI = N холодных стартов, переинициализация контекста |
| **Целевой пользователь** | Разработчик в IDE/CLI | Сервер/CI/SaaS-бэкенд |

## 4. Анализ целесообразности

### 4.1. Когда переход ОПРАВДАН ✅

1. **Headless / серверный сценарий.** Пайплайн нужно запускать не из интерактивной сессии разработчика, а как API-сервис, в CI/CD, по расписанию или по вебхуку — без человека за терминалом. LangGraph здесь профильный инструмент.
2. **Нужны durable-гарантии.** Долгие пайплайны, падения, точное возобновление с конкретного узла, replay для отладки — `interrupt()` + checkpointer решают это строго, тогда как текстовый STOP и `STATE.json` — best-effort.
3. **Жёсткие инварианты контроля.** Лимит fix-loop, обязательность approval-gate, порядок фаз сейчас держатся «на дисциплине модели». В коде графа они становятся структурно невозможными к нарушению.
4. **Промышленная наблюдаемость.** Нужны per-node метрики (токены, латентность, стоимость), трейсы, A/B движков, дашборды — LangSmith/LangGraph дают это из коробки.
5. **Мульти-провайдерность.** Хотите узлы на разных моделях/провайдерах или fallback — граф развязывает оркестрацию от исполнителя.

### 4.2. Когда переход НЕ оправдан / контрпродуктивен ❌

1. **Текущий продукт — это набор для Claude Code.** Его ценность и дистрибуция (`install.sh` в `~/.claude` или проект) завязаны на нативные примитивы. Перенос на LangGraph — это, по сути, **другой продукт**, а не миграция.
2. **Двойная стоимость и холодные старты.** Каждый узел = отдельный запуск CLI → переинициализация system-prompt, правил, MCP, потеря «тёплого» контекста между фазами. Растёт латентность и расход токенов против единой сессии с субагентами.
3. **Дублирование guardrails.** Hooks безопасности (`block-destructive`, `block-secret-writes`, валидация артефактов) придётся либо оставлять в Claude Code (тогда LangGraph их не контролирует), либо переписывать в Python — рост площади поддержки и риск рассинхрона двух источников правды.
4. **Хрупкость сырого CLI.** Парсинг stdout, коды возврата, управление сессиями, передача файлов — источник ошибок. Снимается только переходом на Agent SDK, что фактически возвращает часть нативной логики обратно.
5. **Потеря «zero-code» преимущества.** Сейчас весь пайплайн — читаемый Markdown, правится непрограммистом. LangGraph вводит полноценный Python-проект: зависимости, сборка, тесты, версионирование, деплой.
6. **`Agent` уже даёт оркестрацию.** Делегирование, параллельный review и передача контекста уже работают нативно. LangGraph дублирует то, что платформа предоставляет бесплатно.

### 4.3. Технические риски миграции

| Риск | Серьёзность | Митигация |
|------|-------------|-----------|
| Холодный старт CLI на каждый узел | Средняя | Agent SDK с переиспользованием сессии вместо raw subprocess |
| Рассинхрон guardrails (hooks vs Python) | Высокая | Оставить hooks единственным источником, не дублировать |
| Хрупкий парсинг вывода CLI | Высокая | Использовать `--output-format stream-json` или Agent SDK |
| Рост расхода токенов | Средняя | Кеширование промптов, переиспользование контекста |
| Раздвоение поддержки (2 кодовых базы) | Высокая | Граница: LangGraph только оркеструет, вся логика агентов — в `.claude/agents` |

## 5. Рекомендация

**Гибридный подход, а не замена.** Текущий нативный сетап — лучший выбор для интерактивной разработческой работы и как устанавливаемый продукт; не переписывать его.

LangGraph целесообразен **только если появляется headless/серверный сценарий** (CI, API, расписание) с требованиями durable-исполнения и наблюдаемости. В этом случае:

1. **Не дублировать логику агентов.** Промпты/контракты остаются в `.claude/agents/*.md` и `.claude/rules/*.md`; LangGraph-узлы их лишь загружают и передают.
2. **Вызывать через Claude Agent SDK, а не сырой CLI** — устраняет хрупкий парсинг и холодные старты.
3. **LangGraph отвечает строго за** машину состояний, approval-gate (`interrupt`), fix-loop с жёстким лимитом, fan-out review, checkpointing и трейсинг.
4. **Guardrails оставить в hooks Claude Code** — единственный источник истины для безопасности.
5. Это даёт **второй фронт исполнения** (server runtime) поверх той же базы агентов, а не форк продукта.

**Оценка трудозатрат прототипа** (граф из 8 узлов + approval-interrupt + fix-loop + параллельный review через Agent SDK, без переписывания агентов): порядка **3–5 человеко-дней** до работающего PoC; ещё столько же на паритет по guardrails и наблюдаемости.

### Вердикт
- **Для текущей цели (kit для Claude Code):** перенос НЕ целесообразен — потеря ключевых преимуществ при дублировании уже доступной оркестрации.
- **Для новой цели (серверный/CI runtime):** целесообразен как **дополняющий** слой через Agent SDK с сохранением агентов и hooks как единого источника истины.

Решающий вопрос к заказчику: **где будет запускаться пайплайн — в интерактивной сессии разработчика или как автономный сервис?** Ответ однозначно определяет выбор.
